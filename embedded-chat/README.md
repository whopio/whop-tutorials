# Orbit - a Whop embedded chat demo

Companion demo for the article **"How to add embedded chats to your app or website."** Orbit is a small community app that drops Whop chat into its own pages: a shared channel, a read-only announcements channel, direct messages, and per-user support chats. You act as three different members and chat in real time.

A render-path toggle shows both ways to embed Whop chat: the drop-in `<ChatElement>`, live on a dedicated production company, and the same kind of conversation rendered from the REST Chat API on sandbox. Test cards are not needed here: chat has no checkout.

## How embedded chat works

1. Each of your users is a real Whop account. Orbit creates them as connected accounts with `companies.create` and keeps the returned `owner_user.id`.
2. When a user opens chat, the server mints a short-lived, company-scoped token for that user with `accessTokens.create`. The element (or your own UI) calls this and refreshes it automatically.
3. A channel is a Whop Chat experience. Orbit creates one with `experiences.create` (the built-in Chat app), makes it public, and reads its `chat_feed_` id with `chatChannels.retrieve`.
4. Direct messages and support chats are created with `dmChannels.create` and `supportChannels.create`; both return a `feed_` id.
5. The same element renders any of them: pass the channel id and a token, and Whop handles messages, realtime, media and moderation.

There is no chat database and no websocket server to run. Whop hosts all of it.

## Two ways to render, both live

The **Prebuilt embed** tab is Whop's drop-in `<ChatElement>`, running live on a dedicated production company. The element runtime is production-only, so it cannot authenticate on sandbox; the demo therefore holds a second, production company just for the embed. It is interactive, with profile switching, heavy moderation, and a lazy reset that reseeds the public channel when it goes stale.

The **Chat API** tab renders the same kind of conversation from the REST Chat API on sandbox, with a UI you control (`components/ApiChatPanel.tsx`). Your sends there are simulated locally so the shared sandbox channels stay clean.

The two paths come from separate env + constants: the sandbox company powers the Chat API tab (`WHOP_COMPANY_*`, `constants/whop-ids.ts`), and the production company powers the embed (`WHOP_PROD_*`, `constants/whop-ids.prod.ts`). Leave the `WHOP_PROD_*` vars unset to run on sandbox only; the embed tab then shows a placeholder.

## Key files

Sandbox / Chat API path:

- `lib/whop.ts` - the SDK client, pointed at sandbox or production from `WHOP_SANDBOX`.
- `lib/chat.ts` - server helpers that mint a user token and read/post messages and reactions.
- `app/api/chat/token/route.ts` - mints a company-scoped chat token for a demo user.
- `app/api/chat/messages/route.ts` - reads and sends messages as the acting user.
- `app/api/moderation/route.ts` and `app/api/reset/route.ts` - live channel moderation with `chatChannels.update`.
- `components/ApiChatPanel.tsx` - the live chat rendered from the REST Chat API.
- `constants/whop-ids.ts` - generated by `npm run setup`; the ids for every sandbox surface.
- `scripts/setup-sandbox.mjs` - provisions users, channels, DMs and support chats (idempotent, seeds messages).

Production embed path:

- `lib/prod-env.ts`, `lib/whop-prod.ts`, `lib/prod-chat.ts` - the production company's env, SDK client and Chat API helpers.
- `app/api/chat/prod-token/route.ts` - mints a production chat token for the live element.
- `app/api/prod-reset/route.ts` - wipes and reseeds the public prod channel when stale.
- `components/LiveElement.tsx` / `components/LiveElementSection.tsx` - the real `<ChatElement>` on production, with the profile switcher.
- `constants/whop-ids.prod.ts` - generated by `npm run setup:prod`; the production company's ids.
- `scripts/setup-prod.mjs` - provisions the production demo company (clean handles, heavy moderation, seed).

Shared / article versions:

- `components/ChatDemo.tsx` - the interactive page: render-path toggle, user switcher, surfaces, controls.
- `scripts/setup.mjs`, `components/ChatPanel.tsx`, `components/ChatStage.tsx`, `app/chat/` - the article's minimal versions; `/chat` renders the API path, `/chat/embed` the drop-in element.

## Run it

```bash
npm install
cp .env.example .env.local   # fill in your sandbox key + company id
npm run setup                # provisions sandbox resources, writes constants/whop-ids.ts
npm run dev
```

To also render the live embed, add production credentials (`WHOP_PROD_API_KEY`, `WHOP_PROD_COMPANY_ID`, and a real throwaway `WHOP_PROD_DEMO_EMAIL`) to `.env.local`, then:

```bash
npm run setup:prod           # provisions the production demo company, writes constants/whop-ids.prod.ts
```

Open the printed localhost URL to explore the demo. On the home page, the embed posts for real (moderated, and reset when stale); the Chat API tab's sends are simulated client-side. The article's minimal pages at `/chat` (Chat API) and `/chat/embed` (the element) post for real.

## What this demo deliberately leaves out

- No custom auth. In production you would derive the user id from your own session; here the browser picks which demo user to act as.
- No webhooks or database. State lives in Whop; the app just mints tokens and calls the Chat API.
- The live embed runs on a dedicated throwaway production company, kept behind heavy moderation and a periodic reset, not a real one.
